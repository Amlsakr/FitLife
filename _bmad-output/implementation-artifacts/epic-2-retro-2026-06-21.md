## Epic Review

Epic 2 delivered the full onboarding path:

- OB-001: Welcome level selection
- OB-002: Beginner onboarding path
- OB-003: Intermediate onboarding path
- OB-004: Onboarding completion flag and navigation graph

The epic was structurally successful because each story owned a clean slice of the onboarding journey and the final state stayed within the existing single-activity Navigation 3 shell. By the end, the app could:

- capture the user’s fitness level
- collect beginner or intermediate onboarding data
- persist onboarding completion per authenticated user
- route startup and post-login behavior using the real completion gate

### What Went Well

- The onboarding sequence stayed aligned with the architecture boundary instead of collapsing into `:app`.
- The state-driven MVI pattern held across all four stories, which made the branch logic easier to test and reason about.
- The typed Navigation 3 shell survived the whole epic without needing `NavController`, `NavHost`, or string routes.
- The feature set grew in a stable order: level selection first, then branch-specific data capture, then completion wiring.
- The acceptance criteria for fail-closed behavior were ultimately met in the completion flow and startup gate.

### What Was Hard

- Branch routing and completion logic were more coupled than they first looked. The app had to keep auth verification, onboarding completion, and branch handoff aligned at the same time.
- Review feedback found a few important safety gaps late, especially around unverified sessions, fail-closed startup behavior, and branch-session guard handling.
- Some cleanup issues were not product blockers but still created friction:
- A malformed ViewModel initializer in `MainActivity.kt` surfaced during review.
- A rendered checkmark picked up mojibake and needed a source-level fix.
- The onboarding repository still has broader cancellation and per-user draft-isolation cleanup outside the completion slice.

### Key Learnings

1. Route decisions that affect auth, onboarding, and completion should stay centralized. Once those branches diverge, the risk of one path bypassing another rises quickly.
2. Completion state should be treated as routing metadata with explicit fail-closed behavior, not as a casual boolean.
3. Review coverage needs to include both logic and presentation hygiene. The UI checkmark encoding issue was small, but it was the kind of defect that erodes confidence if it escapes.
4. Keeping the onboarding flow split across stories worked well. Each story remained understandable, and the final completion story could focus on the glue instead of re-litigating branch content.
5. The architecture boundary held up because the team kept the persistence contract in the onboarding feature and adapted it at the app edge.

### Follow-Through From Earlier Work

- Epic 0 established the module graph and MVI/core contracts, and Epic 2 benefited directly from that foundation.
- There was no Epic 1 retrospective available to compare against, so Epic 2 is effectively the first full feature retro in this onboarding sequence.
- The epic exposed a repeated pattern from earlier work: tracker and story hygiene matter almost as much as code hygiene. Several issues were caught in review, not in implementation.

### Technical Debt Carried Forward

- Non-completion onboarding persistence paths still catch `Throwable` and should be hardened to preserve cancellation semantics consistently.
- Selected level and draft persistence remain device-global rather than per-user and should be revisited in a broader onboarding data cleanup.
- The shared Navigation 3 shell still has a root-back behavior gap that can empty the stack in some flows.

## Next Epic Preparation

Epic 3 moves into workout plan generation and the home experience. Based on the planning docs and sprint queue, the next work depends on:

- stable onboarding profile data from Epic 2
- a reliable `users/{uid}` shape for workout planning inputs
- cache-first plan generation with fallback behavior
- protected-flow navigation staying consistent after onboarding completion

### Prep Tasks

- Confirm the onboarding data model exposed to workout plan generation is stable enough for Epic 3 inputs.
- Keep the auth/startup completion gate behavior intact while Epic 3 begins, so onboarding state does not regress under new navigation pressure.
- Carry forward the fallback-first lesson from the Gemini research into the workout plan story set.
- Reuse the typed Navigation 3 shell patterns already proven in Epic 2 rather than inventing a new navigation layer for home or workout flows.

### Epic 3 Dependencies To Watch

- Workout plan generation will depend on the onboarding profile being available and trustworthy.
- The PRD calls for cache-first AI behavior, a 5-second timeout, bounded retry/backoff, and a local fallback plan on failure.
- If onboarding data is inconsistent, Epic 3 will inherit that uncertainty, so any follow-up cleanup on onboarding persistence should be prioritized before or alongside workout plan plumbing.
